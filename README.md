# BioBand AWS — Cloud-Native Health Monitoring Pipeline

A serverless AWS pipeline that rebuilds the backend of **BioBand**, a wearable health monitor originally built as a Rutgers senior capstone project (placed 4th). The original hardware used Firebase for data storage — this project reimagines that pipeline entirely on AWS, from device ingestion to live visualization.

## Why this project exists

The physical BioBand device is no longer available, so this project simulates its sensor output (heart rate, SpO2, temperature) via a Python script standing in for the ESP32/BLE hardware. The focus is the **cloud architecture** — ingesting, storing, archiving, and visualizing IoT sensor data end-to-end on AWS.

## Architecture

```
Python Simulator (generates vitals + anomalies)
        │  publishes via MQTT
        ▼
   AWS IoT Core  ──── Rule ────┐
                                ▼
                        AWS Lambda (store_reading)
                                │
                 ┌──────────────┴───────────────┐
                 ▼                               ▼
           DynamoDB                            S3
     (queryable readings)              (raw JSON archive)
                 │
                 ▼
        AWS Lambda (get_readings)
                 │
                 ▼
           API Gateway (public HTTPS endpoint)
                 │
                 ▼
        Dashboard (HTML + Chart.js)
     live heart rate / SpO2 / temperature charts
```

## AWS Services Used

| Service | Purpose |
|---|---|
| **IAM** | Scoped permissions for the CLI user and each Lambda execution role |
| **IoT Core** | Device identity (certificates), MQTT ingestion, routing rule |
| **Lambda** | Two functions — one to store readings, one to serve them via API |
| **DynamoDB** | Fast, queryable storage of structured sensor readings |
| **S3** | Permanent raw-data archive of every reading received |
| **API Gateway** | Public HTTPS endpoint exposing readings to the dashboard |

## Features

- **Realistic data simulation** — baseline vitals with natural variation, plus injected anomaly events (irregular heart rate, low SpO2, fever-range temperature) to test the pipeline's handling of edge cases
- **Fully automatic pipeline** — no manual triggering; a reading generated by the simulator flows through IoT Core → Lambda → storage without any manual step
- **Dual storage strategy** — DynamoDB for fast queries, S3 for permanent raw archival (a common real-world pattern)
- **Public API + live dashboard** — a browser-based dashboard fetches and charts live data via a public API Gateway endpoint

## Running It

> This project is set up for demonstration purposes and expects an AWS account with the resources described above already provisioned.

1. Run the simulator to generate and publish readings:
   ```
   python3 Simulator/generate_data.py
   ```
2. Readings automatically flow into DynamoDB and S3 via the IoT Core → Lambda pipeline.
3. Open `dashboard/index.html` in a browser to view live charts.

## What I Learned

- End-to-end IoT data pipelines: device simulation, MQTT ingestion, event-driven processing, and storage
- The distinction between DynamoDB (query-optimized) and S3 (cheap, durable archival) and why real systems often use both
- Serverless architecture patterns — Lambda functions triggered by different sources (IoT Core rules vs. HTTP requests via API Gateway)
- IAM least-privilege permissioning across multiple services and roles
- Debugging real permission and configuration issues (IAM roles, CORS, region mismatches) rather than working from a pre-solved tutorial
